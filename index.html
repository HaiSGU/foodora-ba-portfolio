Foodora — Order & Pick-up (MVP)

Problem: Restaurants need a simpler way to take pickup orders (no delivery cost), while customers still expect a clear, low-friction flow from menu to confirmation.

Scope: End-to-end pickup journey (browse menu → cart → checkout → confirmation) with the supporting business rules/constraints needed to make the MVP buildable and testable.

Concrete outputs: A PRD written as epics + user stories with acceptance criteria; 3 user personas; BPMN process flows; ERD; wireflow/screens to communicate the UX; traceability matrix (RTM) to connect requirements → tests; and a validation pack (UAT cases + sign-off) plus SQL scripts for data/logic sanity checks.

Evidence (in deliverables): 14 total deliverables, 5 diagrams, 16 UAT test cases, and 3 UAT documents (summary section on the Deliverables page).

Retail Operations Backoffice

Problem: A retail backoffice has many moving parts (sales, inventory, purchasing, transfers, cashbook). Without clear scope and models, it’s easy to build inconsistent flows and weak reporting.

Scope: Multi-branch operations with the key modules and their end-to-end interactions (what happens, who does it, and what data moves).

Concrete outputs: PRD-lite + user stories to define scope; 3 core system diagrams (context → BPMN → ERD) to align business + data; a module diagram library (use case / sequence / activity) to explain behavior per module; RTM + UAT mini pack for validation; and an SQL showcase pack translating operational questions into example queries/reports.

Design & Testing — MOW Garden

Problem: Testing is hard to evaluate when it’s only “a plan” — reviewers want to see concrete test assets (scenarios, cases, NFR coverage) and how they tie back to the system behavior.

Scope: Design views to explain behavior + a testing pack that demonstrates coverage across functional and non-functional areas.

Concrete outputs: Use cases (XLSX), decomposition + sequence diagrams, screen inventory (XLSX), a test plan (PDF), a multi-sheet test design pack (workflow/functional/UI), a non-functional testing pack (usability/compatibility/security/DB), acceptance & crowd testing sheets, defect/review templates, sample test data, and JMeter performance plans (.jmx) with a performance sheet.

Evidence (in deliverables): The project folder contains 38 files (excluding Detai01), and key Excel packs can be previewed via the in-browser XLSX viewer.

SAP ERP — Order-to-Cash (O2C)

Summary: Simulated BA engagement for an ERP O2C flow with pricing and credit controls (learning-focused: I’m actively studying SAP concepts).

Note: This is a learning project to practice BA artifacts and process thinking — I’m not claiming hands-on SAP configuration experience.

Problem: O2C processes often fail in real life due to unclear scope/rules, unclear role boundaries, and weak validation (requirements don’t map cleanly to tests).

Scope: A simulated O2C scenario focusing on the BA side: scope + NFRs, pricing/credit-related rules, role-based access boundaries, and a validation approach that is traceable and reviewable.

Concrete outputs: Stakeholders + RACI, requirements + scope + NFRs, business rules + decision tables, roles/permissions (RBAC), RTM, UAT test cases (XLSX) plus a UAT summary/sign-off style report (IEEE 829), supporting diagrams (context/process/ERD), and an SQL proof pack.
