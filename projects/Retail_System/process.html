BA Process

How I would run this as a real project (not just a diagram set)

1) Discovery

  • Business goals: reduce stock variance, standardize operations, improve visibility.
  • Users & roles: cashier, inventory staff, warehouse staff, store manager, admin.
  • Constraints: multi-branch data, permissions, auditability, operational speed.

2) Stakeholders & Access Control

A recruiter-friendly improvement is showing you understand roles, permissions, and audit. This project naturally fits a role-based access control model.

  • Branch manager: approvals, overrides, reporting.
  • Cashier: sales-only actions, limited access to pricing/promotions.
  • Inventory staff: stocktake, adjustments with reason codes.
  • Admin: user management, system configuration.

3) Requirements Strategy

  • Functional: defined via UC + sequence + activity (already present).
  • Non-functional: performance, availability, security, usability.
  • Edge cases: partial returns, cancelled transfers, negative stock rules, promo conflicts.
  • Acceptance criteria: written in testable terms (Gherkin-style where helpful).

4) Modeling Approach

  • Use Case: what the user can do (scope + responsibilities).
  • Sequence: how interactions happen over time (services/data updates).
  • Activity: rules + decision points (the process truth table).

Your current assets already cover these 3 views — the upgrade is to make them consistent and traceable.

5) Validation (UAT-ready)

  • Define 1–2 high-value end-to-end scenarios (e.g., Purchase → Stock → Sale → Cashbook).
  • Create a test case set that maps back to UC IDs and activity decisions.
  • Provide a short sign-off summary (pass/fail counts + issues) to feel “real”.
